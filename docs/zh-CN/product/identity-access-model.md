# 用户、组织与角色设计逻辑

:::: info 文档信息
版本：v1.0
更新日期：2026-07-13
::::

## 一句话说明

AGIOne 的访问范围不是只由账号决定，而是由**组织、用户、成员、项目、角色和授权范围**共同决定。

## 核心关系

```text
组织
  -> 定义业务主体和数据边界
用户
  -> 标识登录平台的具体人员
成员
  -> 表示用户在组织或团队中的协作身份和状态
项目
  -> 承载预算、成员协作、API Keys、调用用量和活动记录
角色
  -> 定义用户或成员承担的职责和可见菜单
授权范围
  -> 决定最终可见资源、数据记录和可执行操作
```

简单理解：**组织决定边界，用户负责登录，成员决定协作关系，项目承载业务空间，角色决定职责，授权范围决定具体能看什么、能做什么。**

## 核心对象

| 对象 | 新手理解 | 影响范围 | 用户手册参考 |
| --- | --- | --- | --- |
| 组织 | 平台中的业务主体或客户空间，替代原“租户”概念 | 数据边界、管理员、角色、标签、业务归属和组织可见范围 | [组织](../usermanual/settings/operator/tenants/tenants/) |
| 用户 | 可登录平台的个人账号 | 登录身份、个人操作记录、个人 Key、个人设置 | [设置](../usermanual/settings/) |
| 成员 | 用户加入某个组织、团队或项目后的协作身份 | 成员状态、团队协作、角色分配、项目可见性和操作权限 | [团队成员](../usermanual/settings/operator/members-roles/members/) |
| 项目 | 组织或服务商侧承载模型调用和协作的业务空间 | 项目预算、成员范围、API Keys、用量、活动、模型白名单和设置 | [项目](../usermanual/settings/user/personal/projects/) |
| 角色 | 一组职责和权限集合，例如运营方、模型提供方或普通用户 | 菜单、按钮、任务归属和允许执行的操作 | [角色对比总览](./role-comparison) |
| 授权范围 | 向组织、成员、项目或账号开放的资源、记录和操作 | 可见地域、资源池、云账号、模型、模板、账务记录、License、审计日志和 API 流控规则 | [用户手册](../usermanual/) |

## 访问范围如何生效

```text
登录
  -> 识别用户
  -> 确认所属组织
  -> 确认成员状态
  -> 加载角色权限
  -> 应用项目范围
  -> 应用资源与数据授权
  -> 展示允许访问的菜单、按钮和数据
```

有菜单不代表能看到所有资源；能看到资源也不代表可以执行所有操作。文档中的菜单或对象不可见时，应先检查组织、成员、项目、角色和授权范围，再判断是否属于产品故障。

## 组织、成员、项目的边界

### 组织

- 组织是平台中的业务主体或客户边界。
- 一个组织可以包含多个用户和成员。
- 组织可以拥有管理员、角色、标签、状态和业务身份。
- 跨组织访问必须显式治理，用户不能默认看到其他组织的资源或数据。

### 成员

- 成员是用户加入组织、团队或项目后形成的协作身份。
- 成员状态会影响是否能登录、协作、查看项目或执行操作。
- 成员通常需要结合角色一起判断：同一个用户在不同组织或项目中可以承担不同职责。
- 成员变更可能影响项目协作、调用权限、额度申请、审计记录和团队设置。

### 项目

- 项目是组织或服务商侧承载业务调用的空间。
- 项目通常关联成员、预算、API Keys、模型白名单、调用用量、活动记录和设置。
- Project Key 的可用性不仅取决于 Key 状态，也取决于项目状态、预算、成员范围和模型白名单。
- 项目不可见时，先确认当前用户是否是项目成员，再检查组织、角色和项目状态。

## 默认角色模型

| 角色 | 主要职责 | 典型范围 |
| --- | --- | --- |
| `admin` | 管理平台身份与访问基础 | 组织、用户、成员、角色、菜单和平台级权限配置 |
| `operator` | 准备和治理平台资源、发布内容、财务运营和设置 | 资源接入、模板、授权、配额、监控、基础设置、审核、客户财务、License、审计和 API 流控 |
| `provider` | 提供和运营模型服务 | 发布单模型或聚合模型，维护版本和定价，提交审核，查看客户调用与收益，管理授权范围内的团队、项目和 Key 设置 |
| `enduser` | 使用已授权的平台能力 | 发现和体验模型，调用 API，部署可用服务，查看个人调用、项目、用量、账务或设置 |

角色可见菜单可能随部署版本和平台配置变化，实际入口以当前环境为准。

## 角色与授权不是一回事

| 现象 | 先检查 | 再检查 |
| --- | --- | --- |
| 菜单不可见 | 当前账号是否具有负责该任务的角色 | 当前部署是否向该角色开放菜单 |
| 地域、集群或资源池不可见 | 角色是否可以使用对应子系统 | 组织、业务地域或资源池是否已授权 |
| 模型市场中找不到模型 | 账号是否是普通用户或其他允许的使用方 | 模型审核状态、可见范围和访问权限 |
| 模型提供方无法发布 | 当前账号是否具有 `provider` 角色 | 模型来源、模板、标签和组织权限是否准备完成 |
| 项目不可见 | 当前账号是否是项目成员 | 项目状态、组织范围、成员范围和角色权限 |
| Project Key 不可用 | Key 是否启用，当前账号是否允许使用该项目 | 项目预算、模型白名单、项目状态和 Key 限额 |
| 看不到财务或 License 记录 | 账号是否具备账务、财务、提供方或用户侧所需角色 | 账期、组织、客户、账户、License 范围和同步状态 |
| 看不到设置或 API 流控入口 | 账号是否可执行设置或审计任务 | 组织范围、成员状态、角色菜单配置和规则可见范围 |
| 操作按钮不可见 | 当前角色是否负责该任务 | 按钮权限、资源状态和操作风险控制 |

## 推荐职责划分

| 参与方 | 推荐角色 | 职责边界 |
| --- | --- | --- |
| 部署方平台负责人 | `admin` | 维护组织、用户、成员、角色和平台访问基础 |
| 平台运营、财务、安全或基础设施团队 | `operator` | 准备资源，维护治理配置，审核内容，管理客户财务或 License，维护设置，监控平台运行 |
| 模型或技术团队 | `provider` | 发布和维护模型，提交审核，查看提供方侧调用与收益，维护授权范围内的团队、项目和 Key 设置 |
| 业务用户或应用开发者 | `enduser` | 体验和调用已授权模型，查看个人用量、项目、账务或设置 |

不应为了让缺失菜单出现，就直接分配 `admin` 或宽泛的 `operator` 权限。应围绕任务所需的最小范围调整角色、项目范围或资源授权。

## 访问问题排查

1. 确认当前子系统和目标任务。
2. 确认当前登录用户和所属组织。
3. 确认用户在目标组织、团队或项目中的成员状态。
4. 确认当前角色是否负责该任务。
5. 确认目标项目是否可见、启用，且当前账号属于项目成员。
6. 确认目标地域、资源池、集群、云账号、模型、模板、账务记录、License、设置、审计日志或流控规则已完成授权。
7. 确认运营方已准备相关前置资源。
8. 入口仍然缺失时，向平台管理员提供页面路径、预期入口、账号角色、组织、项目、成员状态和发现时间。

::: details 面向维护人员的实现说明
部分部署会使用指定的平台组织承载 `admin` 和 `operator`，普通组织承载 `provider` 和 `enduser`。成员、项目、角色菜单、用户角色、按钮和 API 权限映射可能存储在实现对象中，并由后端继续校验接口权限。这些名称属于实现细节，不应作为面向客户的主要解释，也不应视为稳定的公开接口。
:::
