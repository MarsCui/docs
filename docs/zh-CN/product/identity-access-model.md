# 用户、租户与角色设计逻辑

:::: info 文档信息
版本：v1.1
更新日期：2026-07-13
::::

## 核心关系

在 AGIOne 中，访问范围由四个相互连接的概念决定：

**租户定义组织边界 -> 用户标识具体人员 -> 角色定义职责 -> 授权决定可见资源和可执行操作**

| 对象 | 新手理解 | 影响范围 |
| --- | --- | --- |
| 租户 | 账号所属的组织或客户空间 | 组织的数据边界和可用资源范围 |
| 用户 | 个人登录平台使用的账号 | 个人身份、操作记录和已分配角色 |
| 角色 | 一组职责，例如运营方或模型提供方 | 菜单、任务归属和允许执行的操作 |
| 授权范围 | 向租户或账号开放的资源、记录和操作 | 可见地域、资源池、集群、云账号、模型、模板、账务记录、License 数据、设置、审计日志、API 流控规则和数据 |

账号需要同时具备正确角色和正确资源授权。有菜单不代表能看到所有资源，能看到资源也不代表可以执行所有操作。

## 访问范围如何生效

```text
登录
  -> 识别租户和用户
  -> 加载已分配角色权限
  -> 应用资源授权范围
  -> 展示允许访问的菜单、按钮和数据
```

文档中的菜单或对象不可见时，应先检查账号角色、租户、授权范围和前置资源，再判断是否属于产品故障。

## 默认角色模型

| 角色 | 主要职责 | 典型范围 |
| --- | --- | --- |
| `admin` | 管理平台身份与访问基础 | 租户、用户、角色、菜单和平台级权限配置 |
| `operator` | 准备和治理平台资源、发布内容、财务运营和设置 | 资源接入、模板、授权、配额、监控、基础设置、审核、客户财务、License、审计和 API 流控 |
| `provider` | 提供和运营模型服务 | 发布单模型或聚合模型，维护版本和定价，提交审核，查看客户调用与收益，管理授权范围内的团队设置 |
| `enduser` | 使用已授权的平台能力 | 发现和体验模型，调用 API，部署可用服务，查看个人调用、用量、账务或设置 |

角色可见菜单可能随部署版本和平台配置变化，实际入口以当前环境为准。

## 租户边界

- 租户代表一个组织或客户边界。
- 一个租户可以包含多个具有不同角色的用户。
- 模型提供方和普通用户可以属于同一租户，但获得不同权限。
- 跨租户访问必须显式治理，用户不能默认看到其他租户的资源或数据。
- 平台级身份治理权限应保留给部署方或获得授权的平台管理团队。

## 角色与授权不是一回事

| 现象 | 角色检查 | 授权检查 |
| --- | --- | --- |
| 菜单不可见 | 确认账号具有负责该任务的角色 | 确认当前部署已向该角色开放菜单 |
| 地域、集群或资源池不可见 | 确认角色可以使用对应子系统 | 确认租户、业务地域或资源池授权 |
| 模型市场中找不到模型 | 确认账号是普通用户或其他允许的使用方 | 确认模型审核状态、可见范围和访问权限 |
| 模型提供方无法发布 | 确认具有 `provider` 角色 | 确认模型来源、模板、标签和租户权限已准备 |
| 看不到财务或 License 记录 | 确认账号具备账务、财务、提供方或用户侧所需角色 | 确认账期、组织、客户、账户、License 范围和同步状态 |
| 看不到设置或 API 流控入口 | 确认账号可执行设置或审计任务 | 确认组织范围、角色菜单配置和规则可见范围 |
| 操作按钮不可见 | 确认当前角色负责该任务 | 确认按钮权限和资源状态 |

## 推荐职责划分

| 参与方 | 推荐角色 | 职责边界 |
| --- | --- | --- |
| 部署方平台负责人 | `admin` | 维护租户、用户、角色和平台访问基础 |
| 平台运营、财务、安全或基础设施团队 | `operator` | 准备资源，维护治理配置，审核内容，管理客户财务或 License，维护设置，监控平台运行 |
| 模型或技术团队 | `provider` | 发布和维护模型，提交审核，查看提供方侧调用与收益，维护授权范围内的团队设置 |
| 业务用户或应用开发者 | `enduser` | 体验和调用已授权模型，查看个人用量、账务或设置 |

不应为了让缺失菜单出现，就直接分配 `admin` 或宽泛的 `operator` 权限。应围绕任务所需的最小范围调整角色或资源授权。

## 访问问题排查

1. 确认当前子系统和目标任务。
2. 确认当前登录用户、租户和角色。
3. 对照[角色对比总览](./role-comparison)确认任务归属。
4. 确认目标地域、资源池、集群、云账号、模型、模板、账务记录、License、设置、审计日志或流控规则已完成授权。
5. 确认运营方已准备相关前置资源。
6. 入口仍然缺失时，向平台管理员提供页面路径、预期入口、账号角色、租户和发现时间。

::: details 面向维护人员的实现说明
部分部署会使用指定的平台租户承载 `admin` 和 `operator`，普通租户承载 `provider` 和 `enduser`。角色菜单、用户角色、按钮和 API 权限映射可能存储在 `sys_role`、`sys_role_menu`、`sys_user_role`、`sys_menu` 等实现对象中，并由后端继续校验接口权限。这些名称属于实现细节，不应作为面向客户的主要解释，也不应视为稳定的公开接口。
:::
