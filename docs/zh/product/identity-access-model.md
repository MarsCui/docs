# AGIOne 平台 — 用户、租户与角色设计逻辑

## 1. 核心关系

AGIOne 里，**租户决定组织边界，用户属于租户，角色决定用户能力，菜单和权限决定用户能看到和操作什么**。

| 对象 | 含义 | 关键规则 |
| --- | --- | --- |
| 租户 | 机构、客户或组织边界 | 一个租户下可以有多个用户 |
| 用户 | 登录平台的个人账号 | 用户通过 `tenantId` 归属到租户 |
| 角色 | 一组能力和权限的集合 | 用户可以绑定一个或多个角色 |
| 菜单/权限 | 页面目录、按钮、接口权限 | 角色绑定后，用户登录才能看到对应菜单并执行对应操作 |

登录后，系统会根据用户绑定的角色加载菜单和权限。用户只能看到被授权的菜单目录；按钮和接口也会继续按权限码校验。

## 2. 租户设计

| 租户类型 | 判断方式 | 默认承载角色 | 能力边界 |
| --- | --- | --- | --- |
| 超管租户 / 运营方租户 | `tenantId = "1"` | `admin`、`operator` | 管平台、管租户、管角色、管菜单、做跨域运营 |
| 普通租户 | 其它 `tenantId` | `provider`、`enduser` | 使用平台能力、提供模型/资源、调用模型服务 |

关键点：

- `operator` 不是独立租户，而是超管租户下的运营用户。
- `provider` 和 `enduser` 都属于普通租户，二者不是靠租户类型区分，而是靠绑定的角色区分。
- 同一个普通租户内，可以同时有 provider 用户和 enduser 用户。
- 面向客户文档建议统一使用 `enduser`，不再使用内部缩写 `eu`。

## 3. 默认角色总览

| 角色 | 所属租户 | 核心定位 | 主要能力 |
| --- | --- | --- | --- |
| `admin` | 超管租户 | 平台超级管理员 | 管理租户、基础角色、菜单、权限等平台元数据 |
| `operator` | 超管租户 | 平台运营管理员 | 模型审核治理、云资源接入、算力资源运营、账务运营 |
| `provider` | 普通租户 | 模型/算力供给和部署使用者 | 创建模型、接入 BYOK、部署模型、管理调用统计、使用算力资源 |
| `enduser` | 普通租户 | 终端使用者 | 浏览模型、体验模型、调用模型、查看用量、管理个人调用记录 |

## 4. admin 能力

| 维度 | 内容 |
| --- | --- |
| 角色定位 | 全平台超级管理员 |
| 主要对象 | 租户、用户、基础角色、菜单、权限元数据 |
| 典型操作 | 创建/管理租户，维护角色，配置菜单，维护权限元数据 |
| 菜单范围 | User Space / 用户空间中的租户、角色、菜单等账号域能力 |
| 权限边界 | 不负责具体业务域的日常模型、算力、账务运营 |
| 客户理解 | 平台总管理员，通常由部署方或平台最高管理方持有 |

## 5. operator 能力

| 维度 | 内容 |
| --- | --- |
| 角色定位 | 平台运营管理员 |
| Model Services | 管理元模型、模型来源、模板、标签、币种；审核模型和应用；管理应用列表 |
| Cross-Platforms / AI Infra-On Cloud | 接入云平台、云平台授权、接入账号、接入资源池、资源池授权；管理模型库、模型分类、推理框架、推理镜像 |
| Heterogeneous-XPUs | 管理资源池、集群、节点、存储、镜像、规格、配额、额度、审批、监控和统计 |
| Trading / Finance | 管理公告、协议、平台客户账户、充值单等运营侧账务能力 |
| 权限边界 | 管平台供给和治理，不作为普通业务用户消费模型 |
| 客户理解 | 平台运营/运维团队角色，负责把模型、算力和平台资源供出来、管起来 |

## 6. provider 能力

| 维度 | 内容 |
| --- | --- |
| 角色定位 | 供给侧角色，也承担部分部署和资源使用能力 |
| Model Services | 创建和管理自有模型；AGIOne 托管部署；第三方 BYOK 接入；多模型聚合；查看客户调用概览、调用日志和调用统计 |
| Cross-Platforms / AI Infra-On Cloud | 接入个人云账号；使用模型服务快速开始；浏览模型广场；管理我的部署 |
| Heterogeneous-XPUs | 创建开发环境、训练任务、推理任务；管理实例、存储、镜像、数据集、创作空间等资源 |
| 权限边界 | 不应拥有平台级租户、全局菜单、全局资源治理权限 |
| 客户理解 | 适合负责模型接入、模型发布、云账号接入、算力部署和资源维护的团队或人员 |

## 7. enduser 能力

| 维度 | 内容 |
| --- | --- |
| 角色定位 | 终端使用者 |
| Model Services | 浏览模型市场；使用文本、图像、语音、视频体验中心；查看我的调用、调用日志、调用统计；查看用量与收益；使用私有模型或聚合模型 |
| Finance / Deck | 账务充值、积分扣减等消费侧能力 |
| 权限边界 | 不应拥有模型审核、云资源接入、平台级菜单和租户治理权限 |
| 客户理解 | 适合只需要使用、体验、调用模型能力的业务人员 |

## 8. 自定义角色与菜单权限

AGIOne 支持按需创建自定义角色。实际控制逻辑是：

```text
创建角色
  -> 给角色绑定菜单、按钮、API 权限
  -> 给用户绑定角色
  -> 用户登录后只看到对应菜单，并只能执行对应操作
```

| 配置动作 | 对应对象 | 作用 |
| --- | --- | --- |
| 创建角色 | `sys_role` | 定义一组业务能力；`type=2` 表示租户自定义角色 |
| 配置菜单权限 | `sys_role_menu` + `sys_menu` | 决定角色能看到哪些菜单、按钮和 API 权限 |
| 绑定用户 | `sys_user_role` | 决定某个用户获得哪些角色能力 |
| 菜单控制 | `sys_menu.menuType=0` | 控制左侧目录和页面入口 |
| 按钮控制 | `sys_menu.menuType=1` + `permission` | 控制新增、编辑、删除、导入、导出等按钮 |
| 接口控制 | `@HasPermission` + `sys_menu.permission` | 控制后端接口是否允许调用 |

一句话给客户解释：

> 平台不是按用户逐个开菜单，而是按角色授权。角色绑定了哪些菜单和权限，用户登录后就只能看到和操作这些内容。

## 9. 企业私有化场景示例

如果一家企业部署 AGIOne，并提供给其他企业或外部组织人员使用，可以先按下面方式理解：

| 参与方 | 推荐角色 | 能力边界 |
| --- | --- | --- |
| 部署方企业平台管理员 | `admin` | 管租户、管角色、管菜单、管平台基础权限 |
| 部署方企业运营/运维团队 | `operator` | 管模型审核、资源接入、算力供给、账务运营 |
| 被服务企业模型/技术团队 | `provider` 或 provider 派生角色 | 接入模型、接入云账号、部署模型、查看调用统计 |
| 被服务企业业务使用人员 | `enduser` 或 enduser 派生角色 | 调用模型、体验模型、查看个人调用和用量 |

平台级治理权限建议留在部署方企业；被服务企业获得哪些能力，应通过租户、角色和菜单权限进行精细授权。
